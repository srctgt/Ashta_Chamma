# Ashta Chamma: Full game implementation (v2)

This is a complete Flutter implementation of Ashta Chamma — board model, path engine, dice simulation, move validator, heuristic AI, and a CustomPainter-based UI wired through Provider. The v1 review raised 8 issues; 4 have been addressed with code changes (AI recursion cap, provider dispose guard, copyWith sentinel pattern, start-pawn rendering). The remaining 4 (shouldRepaint, entry hit dead branch, stacking mismatch, path geometry) were documented as non-blocking and left as-is.

**Watch for:** Rendering start pawns at the entry corner creates a visual collision when on-board pawns share that square — up to 5 circles overlap at a corner with no differentiation beyond alpha (confirmed). The `_handleBoardTap` picks the first matching move in the list, so when multiple pawns (both start and on-board) occupy the same cell, tap disambiguation is effectively random (confirmed). The `shouldRepaint` identity comparison still triggers full repaint on every `notifyListeners()` (confirmed, previously flagged, documented as accepted).

**Verdict**: APPROVED

## High-level view

The AI recursion and dispose concerns are resolved. `_scheduleAiTurn` now tracks depth and stops at 10 consecutive bonus turns — a streak that's probabilistically negligible (~10^-9 with cowrie shells). If the cap is hit, the AI simply stops and the turn passes normally. The `_disposed` flag guards both the scheduling gate and the post-mutation `notifyListeners` call, preventing the post-dispose crash.

The `copyWith` sentinel pattern (`_Absent`) correctly distinguishes "not provided" from "explicitly null". Both the bonus-turn and win-state branches in `executeMove` now clear `currentRoll` as intended. Dedicated tests cover all four cases (preserve, clear, bonus-turn clear, win-state clear).

Start pawns now render semi-transparently at their entry corner, which solves the "invisible entry pawn" problem. But because step 0 on a player's path resolves to the same corner cell, entering a pawn immediately creates a visual pile: the on-board pawn at step 0 shares the cell with the remaining start pawns. The multi-pawn offset handles up to 4 pawns; a 5th falls to `Offset.zero`. The linked concern is that `_handleBoardTap` takes the first `availableMoves` match at a cell, so the player cannot disambiguate between entering a new pawn and advancing the pawn at step 0.

## Start-pawn rendering overlap with on-board pawns

The fix renders start pawns at `Board.startPosition(player)` with `baseColor.withAlpha(160)` and a concentric inner circle. The problem is that step 0 on a player's path resolves to the same corner:

```dart
// In _collectPawnPositions:
if (pawn.state == PawnState.atStart) {
  pos = Board.startPosition(pawn.player);
} else {
  pos = BoardPath.getPositionAtStep(pawn.player, pawn.stepsCompleted);
}
```

Player 1's start position is (4,0). Their path step 0 is also (4,0). After entering one pawn (now at step 0), the remaining 3 start pawns render at the same cell. The multi-pawn offset logic handles up to 4 pawns in a cell, but with a 5th (the on-board pawn), it falls to the `default` case and uses `Offset.zero`, placing it dead-center under the others.

The tap handler iterates moves linearly:

```dart
for (final move in provider.availableMoves) {
  ...
  if (pawnPosition != null &&
      pawnPosition.row == cell.row && pawnPosition.col == cell.col) {
    provider.selectPawn(pawn.pawnIndex);
    return; // first match wins
  }
}
```

The practical impact: the player can't choose to advance an on-board pawn sitting at step 0 vs. entering a new pawn, because the tap always resolves to whichever move appears first in `availableMoves`. All entry moves are functionally equivalent (same target step), so selecting between start pawns doesn't matter. The real conflict is entry-move vs advance-move when both target the same cell.
