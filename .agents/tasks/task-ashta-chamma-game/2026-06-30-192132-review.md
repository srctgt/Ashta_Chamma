# Ashta Chamma: Full game implementation from scratch

This change creates an entire Flutter-based Ashta Chamma (Chowka Bhara) board game from zero. The implementation spans a game logic engine (board model, path calculation, dice/cowrie simulation, move validation, game state machine), a heuristic AI opponent, and a full UI layer using CustomPainter for the board with Provider-based state management.

**Watch for:** AI bonus turn recursion has no depth cap and can stack overflow in degenerate cases (confirmed). The `GameState.copyWith` cannot clear `currentRoll` or `winnerId` back to null once set (confirmed). Board path traversal direction may not match traditional anti-clockwise movement as specified (likely). The `_scheduleAiTurn` in `GameProvider` fires without lifecycle guards and can mutate state after the widget is disposed (confirmed).

## High-level view

The board model defines a 16-position outer ring and 8-position inner ring traversed as a flat step index (0-24) per player. The path direction is implemented as decrementing indices through a clockwise-defined ring array, which achieves anti-clockwise movement, but the inner ring entry points for each player determine whether the full path geometry matches the traditional game's expected routes.

The game state machine is immutable and transition-based: `rollDice()` produces a new state with available moves computed eagerly, and `executeMove()` resolves hits, bonus turns, and win conditions in a single transition. A gap exists where `copyWith` cannot null out fields due to Dart's default-to-existing semantics, causing stale roll values to persist into game-over state.

The AI scores moves with a static point system (hit=100, home=90, entry=40, progress scaled to 50). The vulnerability bonus iterates all opponent pawns across all possible dice values (1-8), making it O(pawns * 8) per move scored — fine for 4 pawns, but the recursive AI turn loop in the provider has no termination guard beyond the game-over check.

The UI layer relies on `CustomPaint` for board rendering and a `GestureDetector` on the paint area for tap-to-cell mapping. `BoardPainter.shouldRepaint` compares `GameState` and `List<Move>` by reference identity, which means every `notifyListeners()` call triggers a full repaint regardless of whether visual state changed. Pawns at start are not rendered on the board, leaving players with no visual target to tap when entering pawns.

1. **AI turn infinite recursion** — `_scheduleAiTurn` calls `performAiTurn`, which calls `_scheduleAiTurn` again on bonus turns. No depth limit exists; a streak of bonus rolls (4 or 8) will stack `Future.delayed` callbacks indefinitely. Add a `consecutiveBonusTurns` cap or iteration limit in the provider.
2. **Provider fires after dispose** — `_scheduleAiTurn` uses `Future.delayed` without checking whether the `ChangeNotifier` has been disposed. Navigating away mid-AI-turn will throw. Guard with a `_disposed` flag or cancel pending futures on `dispose()`.
3. **copyWith cannot null fields** — `GameState.copyWith` uses `??` for `currentRoll` and `winnerId`, making it impossible to reset them to null. The `executeMove`'s bonus-turn and win-check branches pass `currentRoll: null` through `copyWith`, which silently keeps the old value. Fix by using a sentinel or nullable wrapper pattern.
4. **Outer ring traversal direction** — The outer ring list is defined top-left clockwise, and `getPlayerPath` walks it with decrementing indices. Player 1 at index 12 (4,0) moves toward index 11 (4,1), 10 (4,2), etc. Verify this matches the intended anti-clockwise game direction for both players.
5. **Entry pawns invisible to tap** — Pawns at start are not rendered on the board, but entry moves require tapping the corner square. Players have no visual target for entry moves. Render start-area pawns or add an entry prompt.
6. **Entry hit dead branch** — `_validateEntry` checks `isSafeSquare` and blocks hits, but the entry step is always a corner (always safe). This branch can never fire. Not a bug, but masks future logic errors if start positions change.
7. **shouldRepaint always true** — `BoardPainter.shouldRepaint` compares by identity. Since `GameState` has no `==` override, every `notifyListeners` triggers a full repaint. Consider a generation counter or `==` on `GameState`.
8. **Validator/painter stacking mismatch** — `_isOwnPawnAtPosition` blocks friendly stacking, but `_drawMultiplePawns` handles up to 4 same-cell pawns. Dead rendering path now, potential drift if stacking rules change.

## Board path geometry and player routing

The outer ring is defined as 16 positions starting from (0,0) going clockwise: top row left-to-right, right column top-to-bottom, bottom row right-to-left, left column bottom-to-top. Player paths traverse this array in reverse index order to achieve anti-clockwise movement. Player 1 begins at index 12 (4,0) and moves to indices 11, 10, 9... wrapping to 15, 14, 13, arriving back at 12 after 16 steps.

The inner ring entry is hardcoded: Player 1 enters at inner index 0 (1,1), Player 2 at inner index 4 (3,3). The comment states Player 2's last outer position before transition is (1,4) at outer index 5, adjacent to (1,3) at inner index 2. But the code uses inner index 4 at (3,3), which is diagonally opposite Player 2's start at (0,4). This contradicts the comment's adjacency rationale. The test suite validates path lengths and continuity but doesn't verify geometric correctness against the traditional game layout — a test that traces both players' physical board positions would catch this if it's wrong.

## AI bonus turn recursion

```dart
void _scheduleAiTurn() {
  Future.delayed(const Duration(milliseconds: 500), () {
    if (_controller == null || isGameOver) return;
    performAiTurn();
  });
}

void performAiTurn() {
  if (_controller == null || !isAiTurn || isGameOver) return;
  _controller!.performAiTurn();
  notifyListeners();
  if (isAiTurn && !isGameOver) {
    _scheduleAiTurn();
  }
}
```

Each bonus turn schedules another delayed callback. With cowrie shells, rolling 4 or 8 (combined 12.5% probability per roll) each grant a bonus turn. The `GameState` tracks `consecutiveBonusTurns` but nothing in the provider or controller reads it to break the cycle. A runaway streak queues unbounded delayed callbacks, each calling `notifyListeners()`. If the user navigates away during this process, the disposed ChangeNotifier throws on `notifyListeners()`.

## State immutability gap in copyWith

```dart
GameState copyWith({
  DiceResult? currentRoll,
  int? winnerId,
  ...
}) {
  return GameState(
    currentRoll: currentRoll ?? this.currentRoll,
    winnerId: winnerId ?? this.winnerId,
    ...
  );
}
```

Passing `currentRoll: null` to `copyWith` keeps the old value. The `executeMove` method calls `copyWith(currentRoll: null, availableMoves: [])` in the bonus-turn path, intending to clear the roll for the next turn. But `null ?? this.currentRoll` preserves the stale roll. The normal turn-switch works because `_switchTurn()` constructs a fresh `GameState` directly, bypassing `copyWith`. The win-detection branch also uses `copyWith` with `currentRoll: null`, so the game-over state retains the last roll value. The UI displays this stale roll on the win screen.

## Tap detection gap for entry moves

When the game is in the moving phase and a pawn has an entry move, the game screen maps taps to the start position:

```dart
if (move.isEntry) {
  pawnPosition = Board.startPosition(pawn.player);
}
```

The player must tap the corner square where the pawn would enter. But the board painter skips rendering pawns with `PawnState.atStart`. A player rolling 4 or 8 with all pawns at start sees no highlighted pawn and must know to tap the corner with no visual affordance. This makes the game effectively unplayable for first-time users without external instructions.

## Test coverage gaps

Not tested: the recursive AI turn scheduling in `GameProvider` (no async/timer tests), the `GameController.performAiTurn` bonus turn loop, the tap-to-move mapping in `GameScreen._handleBoardTap`, navigation between screens, Provider disposal during active AI turns, or the `copyWith` null-field issue. The AI tests verify scoring determinism but don't test behavior under repeated consecutive invocation.
